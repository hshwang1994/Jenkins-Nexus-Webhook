# 이천 운영 Nexus 원본 OOB 파일 동기화 — 확정 설계서

## 1. 목적

포털을 통해 업로드된 파일을 **이천 운영 Nexus** 에 저장한 뒤, 운영망/개발망의 OOB Agent 로컬 경로에 자동 동기화한다. 운영자가 별도 작업 없이 OOB망 작업(BIOS, 펌웨어, 이미지 배포 등)에서 즉시 파일을 사용할 수 있도록 한다.

## 2. 범위

- 원본: **이천 운영 Nexus 1대**
- 동기화 대상: 운영/개발 Master 하위 **OOB Agent (3개 리전 × 운영/개발 = 12대)**
- 동기화 제외: Service Agent, 다른 리전 Nexus

## 3. 핵심 원칙

| # | 원칙 |
|---|---|
| 1 | 원본은 이천 운영 Nexus 단 하나다. |
| 2 | Nexus → 운영 Master, Nexus → 개발 Master 로 각각 직접 Webhook 을 보낸다. 중간 분기 시스템은 두지 않는다. |
| 3 | 각 Master 는 자기 하위 OOB Agent 만 본다. |
| 4 | 리전(`ich`/`chj`/`yi`) + OOB(`redfish`) 라벨 조합으로 후보군을 산출한다. |
| 5 | 대표 OOB 에이전트는 매 실행 시점에 동적으로 선정한다. 고정 역할 라벨은 사용하지 않는다. |
| 6 | 대표 OOB 에이전트가 Nexus 에서 1회 다운로드 후, 같은 리전 대상 OOB 에이전트로 fan-out 한다. |
| 7 | size + checksum 검증 통과 전에는 절대 최종 경로를 덮어쓰지 않는다. |
| 8 | 삭제는 즉시 삭제하지 않고 quarantine 격리 후, 전체 점검이 Nexus 기준 재확인하고 최종 삭제한다. |
| 9 | 주기 점검(부분 30분 / 전체 1일) + 상태 점검(15분) 으로 누락, catch-up, 정합성을 보장한다. |
| 10 | 라벨/경로/스케줄/제외 노드는 `sync_config.yml` 한 곳에서 관리한다. |
| 11 | Nexus 인증은 Jenkins Credentials 에 저장된 읽기 전용 서비스 계정으로만 한다. 코드/설정에 평문 비밀번호 금지. |
| 12 | Webhook 신뢰는 HMAC-SHA256 서명으로 검증한다. |
| 13 | 모든 로그는 한글 5단(제목/발생상황/원인/문제/조치방법)으로 작성한다. |
| 14 | 모든 변수명은 스네이크 케이스로 통일한다. |

## 4. 토폴로지

```
                  [이천 운영 Nexus]
                    │           │
          (Webhook POST)       (Webhook POST)
                    ▼           ▼
       [운영 Master]            [개발 Master]
        ├── ich-oob-01,02       ├── ich-oob-01,02 (개발)
        ├── chj-oob-01,02       ├── chj-oob-01,02 (개발)
        └── yi-oob-01,02        └── yi-oob-01,02 (개발)
```

운영 Master 와 개발 Master 는 독립 소비자다. 한쪽이 실패해도 다른 쪽은 계속 진행한다.

## 5. Webhook 수신 방식

| 항목 | 값 |
|---|---|
| Jenkins 수신 플러그인 | Generic Webhook Trigger Plugin |
| endpoint | `POST http://<jenkins-master>/generic-webhook-trigger/invoke?token=oob-sync-nexus-webhook` |
| Nexus Webhook 구성 | Repository Webhook Capability 를 **운영 Master 용 1개, 개발 Master 용 1개** 총 2개 등록한다. |
| 서명 헤더 | `X-Nexus-Webhook-Signature` |
| 서명 알고리즘 | HMAC-SHA256 (raw body 전체를 secret key 로 서명) |
| 서명 secret 저장 | Jenkins Credentials `nexus-webhook-hmac-secret` (secret text) |
| payload 파싱 | Generic Webhook Trigger 의 `genericVariables` 로 JSON path 기반 추출 |
| 보안 설정 | `printContributedVariables: false`, `printPostContent: false` |

> Nexus 3 CE 의 Repository Webhook Capability 는 repository 당 여러 개 등록이 가능하다. 운영 Master 와 개발 Master 각각에 capability 를 1개씩 할당하여 독립적으로 관리한다.

## 6. 대표 OOB 에이전트 선정 규칙

**적용 전 단계:** `nodesByLabel` 후보 조회 → online 필터 → `exclude_node_names` 제거

| 우선순위 | 기준 | 설명 |
|---|---|---|
| 1순위 | 파일 보유 | 해당 파일을 로컬에 이미 가진 노드를 우선 선정한다. |
| 2순위 | 최근 성공 이력 | `/var/lib/oob-sync/last_sync_state.json` 이 존재하는 노드를 우선 선정한다. |
| 3순위 | 이름 정렬 | 동률일 때만 이름 사전순으로 해소한다. |

## 7. 라벨 체계

| 종류 | 값 |
|---|---|
| 리전 라벨 | `ich`, `chj`, `yi` |
| OOB 선별 라벨 | `redfish` |
| 후보군 selector | `<location_code> && redfish` |

## 8. 경로 규약

| 용도 | 경로 |
|---|---|
| 최종 경로 | `/sync/files/<repository_name>/<asset_path>` |
| 대표 OOB 임시 경로 | `/sync/tmp/<repository_name>/<asset_path>.part` |
| 대상 OOB 임시 경로 | `/sync/incoming/<repository_name>/<asset_path>.part` |
| 삭제 격리 경로 | `/sync/quarantine/<repository_name>/<asset_path>` |

## 9. 스케줄

| 작업 | 주기 |
|---|---|
| 실시간 동기화 | 이벤트 기반 (Webhook) |
| 부분 점검 | 30분마다 |
| 전체 점검 | 하루 1회 (03시 부근) |
| 상태 점검 | 15분마다 |

## 10. 부분 점검 범위

부분 점검은 전체 repository 전수 조사가 아니다. 아래 3가지만 대상으로 한다.

1. **최근 변경분**: Nexus REST API 로 전체 asset 목록을 가져온 뒤, `lastModified` timestamp 를 기준으로 최근 변경분만 후처리 필터링한다. Nexus 3 CE 의 Search API 는 서버 측 시간 필터를 지원하지 않으므로, 클라이언트 측에서 필터링하는 방식을 사용한다.
2. **최근 실패분**: `/var/lib/oob-sync/recent_failures.json` 에 기록된 파일
3. **신규 OOB Agent catch-up**: 새로 편입된 OOB Agent 의 누락 파일

## 11. 신규 OOB Agent catch-up

- catch-up 주체는 **주기 점검(reconcile)** 이다.
- reconcile 은 Nexus 기대 상태를 기준으로 대표 OOB 에이전트의 `/sync/files` 를 먼저 완성한다.
- 그 후 대표 OOB 에이전트가 rsync 로 각 peer 에 전파한다.
- 신규 노드는 `/sync/files` 가 비어 있으므로 rsync 가 모든 파일을 자동 복사한다.

## 12. 인증/계정

| 용도 | Credentials ID | 타입 |
|---|---|---|
| Webhook HMAC 검증 | `nexus-webhook-hmac-secret` | secret text |
| Nexus 다운로드/조회 | `nexus-oob-sync-readonly` | username/password |
| OOB Agent 간 fan-out | `oob-sync-ssh-key` | ssh user private key (username 은 `sync_config.yml` 의 `fanout.ssh_user` 로 지정. 기본값: `cloviradmin`) |

- 운영/개발 Master 양쪽에 동일한 Credentials ID 로 등록한다.
- read + browse 권한만 부여. write/delete/admin 금지.
- 기존 Nexus 계정이 read + browse 권한을 가지고 있으면 재사용 가능하다. 전용 계정이 없을 때만 신규 생성한다.
- Jenkins `withCredentials` 블록 안에서만 사용하며 콘솔 로그 자동 마스킹.

## 13. checksum 기준

- checksum 검증은 **sha256 을 우선 사용**하고, sha256 을 사용할 수 없는 경우 **sha1 으로 fallback** 한다.
- Nexus 3 CE 는 sha1 을 기본 제공하며, 설정에 따라 sha256 도 제공한다.
- Webhook payload 에서 sha256 이 존재하면 sha256 을 사용하고, 없으면 sha1 을 사용한다.
- `sync_config.yml` 의 `checksum.preferred_algorithm` (sha256) 과 `checksum.fallback_algorithm` (sha1) 으로 제어한다.
