# 테스트 시나리오

각 시나리오는 **사전 조건 / 실행 절차 / 기대 결과 / 검증 방법** 을 포함한다.

---

## TC-01. 업로드 실시간 동기화 성공

**사전 조건**: 운영/개발 Master 정상, 모든 OOB Agent online, 대상 repository 존재.

**실행 절차**: 포털에서 200 MB 파일 업로드 → Nexus webhook 발사.

**기대 결과**: 운영/개발 양쪽 동기화 SUCCESS. 12대 모두 `/sync/files/<repo>/<asset>` 에 동일 파일 존재.

**검증 방법**: 각 OOB 에서 `sha256sum` (또는 `sha1sum` fallback) 결과가 Nexus 메타와 동일.

---

## TC-02. 삭제 실시간 동기화 성공

**사전 조건**: TC-01 완료.

**실행 절차**: 동일 파일 삭제 → delete webhook 발사.

**기대 결과**: 모든 OOB 의 파일이 quarantine 으로 이동. 전체 점검 후 최종 삭제.

**검증 방법**: `/sync/files` 에서 사라짐 확인. `/sync/quarantine` 에서 존재 확인 → 전체 점검 후 사라짐 확인.

---

## TC-03. 운영/개발 Master 독립성

**사전 조건**: 개발 Master 만 stop.

**실행 절차**: 파일 업로드.

**기대 결과**: 운영 Master SUCCESS, 개발 Master 미수신. 개발 Master 복구 후 다음 부분 점검에서 catch-up 성공.

---

## TC-04. 대표 OOB 에이전트 선정 정상 동작

**사전 조건**: ich-oob-01 = online (파일 미보유), ich-oob-02 = online (파일 보유).

**실행 절차**: 동기화 트리거 → 빌드 로그에서 `selected_seed_node`, `selection_reason` 확인.

**기대 결과**:
- `selected_seed_node` = `ich-oob-02` (1순위: 파일 보유 기준)
- `selection_reason` 에 "파일을 이미 보유한 노드" 포함

**추가 검증**: 양쪽 모두 파일 미보유 + 양쪽 모두 state 파일 존재 → 이름 정렬 tie-break.

---

## TC-05. 일부 대상 OOB 에이전트 실패 후 재전송

**사전 조건**: ich-oob-02 SSH 일시 차단.

**실행 절차**: 동기화 트리거 → ich-oob-02 SSH 복구 → 부분 점검 대기.

**기대 결과**: 1차 ich-oob-01 성공, ich-oob-02 실패 (UNSTABLE). 부분 점검 후 ich-oob-02 catch-up.

---

## TC-06. 신규 OOB Agent 자동 편입

**사전 조건**: ich-oob-03 신규 등록 (라벨 `ich`, `redfish` 부착).

**실행 절차**: 다음 부분 점검 대기.

**기대 결과**: ich-oob-03 의 `/sync/files` 가 기존 파일들로 채워짐. 이후 실시간 동기화에 자동 포함.

---

## TC-07. 주기 점검이 누락 파일 복구

**사전 조건**: ich-oob-02 의 `/sync/files/<repo>/<asset>` 임의 삭제.

**실행 절차**: 부분 점검 트리거.

**기대 결과**: 누락 파일 복구. 한글 로그에 누락 파일 경로 표시.

---

## TC-08. quarantine 삭제 흐름 검증

**사전 조건**: TC-02 직후.

**실행 절차**: 전체 점검 트리거.

**기대 결과**: quarantine 파일이 Nexus 기준 재확인 후 최종 삭제. 한글 로그 출력.

---

## TC-09. Jenkins Credentials 기반 Nexus 인증

**사전 조건**: `nexus-oob-sync-readonly` credentials 등록 완료.

**실행 절차**: 동기화 트리거 → 콘솔 로그 확인.

**기대 결과**: 평문 비밀번호가 로그에 노출되지 않음. Nexus 401/403 없음.

**검증 방법**: 로그에서 비밀번호 노출 없음 확인.

---

## TC-10. 빌드 보관 정책 동작

**사전 조건**: master_realtime_sync 60회 실행.

**기대 결과**: 최근 50개만 남고 이전 빌드 자동 삭제. 14일 초과 빌드 자동 정리.

---

## TC-11. Webhook HMAC 위조 거부

**실행 절차**: 잘못된 서명으로 webhook_dispatch 호출.

**기대 결과**: HMAC 검증 실패 한글 로그 출력. master_realtime_sync 트리거되지 않음.

---

## TC-12. 20 GB 대용량 업로드

**사전 조건**: 충분한 디스크, 25G OOB 회선.

**기대 결과**: 실시간 동기화 성공. size + checksum 검증 통과. fan-out 중단 시 rsync `--partial` 로 재개.
